I’ll ground this in what you’ve already described for Link-Up, then check the current PWA/platform reality so the recommendation isn’t based on stale browser assumptions.

The important constraint I’m seeing is not “which GUI toolkit is best.” It is distribution plus trust: users must be able to open it instantly, install it casually, receive notifications, and trust that location, identity, photos, blocks, and reports are handled correctly.

My recommendation: make Link-Up a PWA-first product, with optional native shells later.

Not Tauri-first. Not desktop-first. Not pure native. Not pure P2P. The core product should be a mobile-first web app that works as:

PWA in browser → installed PWA → Capacitor native wrapper later

That matches the actual goal: one app surface across mobile web, desktop web, and app-store distribution, without rewriting the product three times. PWAs are now a serious distribution model: they can be installable, launch like apps, use a manifest/service worker, and on iOS/iPadOS Home Screen web apps can use Web Push and badging. [MDN Web Docs]

Sniffies is the right inspiration. The important lesson is not “copy Sniffies.” The lesson is that an adult/gay/geolocation-heavy social product can build massive reach from the web first. Public records describe Sniffies as a geolocation PWA platform serving the gay male market since at least 2017, and later reporting says the native iOS app came after the web platform had already become the foundation. [WIPO]

For Link-Up, I would use this stack:

Frontend: SvelteKit, JavaScript, CSS, PWA.

This is the one place I would move away from your current “vanilla only” instinct. Vanilla JS is fine for a small app, but Link-Up is not a small app. You will have maps, profiles, chat, modals, media uploads, presence, filters, settings, onboarding, moderation flows, payments, notifications, and probably multiple account/privacy modes. That gets ugly fast without a component model.

Svelte is the cleanest fit for your preferences because it keeps the mental model close to HTML/CSS/JS rather than turning everything into framework ceremony. Svelte’s own docs describe it as using HTML, CSS, and JavaScript through a compiler, and SvelteKit is its app framework for building performant web applications. [Svelte]

I would not use React. I would not use Flutter. I would not use React Native. I would not start with Swift/Kotlin. Those choices make the codebase less aligned with your strengths.

Native wrapper: Capacitor, later.

Do not use Tauri as the main direction for this project. Tauri is interesting for desktop packaging, but Link-Up’s first-class target is mobile. Capacitor is specifically built around taking HTML/CSS/JS apps to iOS, Android, and PWA targets from one codebase. [Capacitor]

The native shell should exist only when you need app-store presence, better notification onboarding, camera/media affordances, native purchase APIs, or platform-specific integrations. The PWA should remain the real app.

Backend: Go + PostgreSQL/PostGIS + object storage.

This is where I would be stricter. For an app trying to replace gay dating apps, the backend is not incidental. It is the product’s trust layer.

Use Go for the long-running server side: auth, profiles, geospatial queries, WebSocket presence, messaging coordination, reports, blocks, moderation queues, push notification fanout, abuse controls, rate limits, and payments. Use PostgreSQL with PostGIS for geospatial data. Use S3-compatible object storage for photos/videos. Use Redis or Valkey only where you actually need ephemeral state, queues, rate limits, or presence acceleration.

Bun is still useful for tooling, scripts, frontend builds, and prototypes. I would not make Bun the long-term production backend bet for this app unless keeping one language matters more than operational predictability.

Map layer: MapLibre GL JS.

For the map UI, use MapLibre GL JS. It is open-source and built for interactive vector maps in the browser using WebGL. Do not depend on free OpenStreetMap tile servers at scale; OSM data is free, but the public tile servers are donation-funded and capacity-limited. Use a paid tile provider early or plan to host/cache tiles properly. [MapLibre]

Local-first: yes for UX, no for authority.

This is the correction I would make to the earlier local-first/P2P direction.

Use local-first behavior for speed, resilience, drafts, cached profiles, recent chats, offline reads, optimistic UI, and maybe encrypted local storage. IndexedDB is the right browser primitive for significant structured client-side storage, including blobs/files. [MDN Web Docs]

But do not make the first version truly decentralized or P2P-authoritative. Dating apps need central authority for blocks, reports, bans, trust, profile visibility, location privacy, abuse prevention, rate limits, account recovery, and legal compliance. P2P can come later for direct media transfer, calls, or private channels. WebRTC data channels can do peer-to-peer bidirectional data transfer, but that should not be the foundation of discovery, identity, or moderation. [MDN Web Docs]

Privacy model: approximate location by design.

Do not build Link-Up around exact live tracking. Build around your geocell idea: approximate presence, density-aware regions, user-controlled precision, and strict separation between “where I am” and “what others can infer.”

Browser geolocation requires user permission and secure contexts, so the UX should be explicit: “show me nearby people now,” not “track me forever in the background.” [MDN Web Docs]

Moderation is not optional.

This matters more than the framework. If Link-Up has adult dating, photos, chat, location, and anonymous/semi-anonymous users, then moderation, blocking, reporting, age gating, non-consensual image handling, spam controls, and safety workflows need to exist from the beginning.

Apple’s guidelines are especially relevant here: apps with user-generated content that become primarily pornographic, random/anonymous chat, objectification, threats, or bullying do not belong on the App Store. Google Play also restricts apps containing or promoting sexually gratifying content and has specific rules around incidental sexual content in UGC apps. [Apple Developer]

So the practical recommendation is:

Build Link-Up as a SvelteKit PWA with plain JavaScript/JSDoc, backed by Go + PostgreSQL/PostGIS, with MapLibre for maps, IndexedDB for local cache, Web Push for notifications, and Capacitor only when you are ready to ship native shells.

Keep the product web-native. Keep the UI simple and fast. Keep the backend authoritative. Keep P2P as an enhancement, not the foundation. That gives you the best chance of building something that can actually compete with existing gay dating apps instead of becoming an architecture experiment.

That exact URL model does not really work in a normal browser.

https://link-up.com/app means scheme https, host link-up.com, and usually port 443. The browser treats that as a single origin based on scheme, host, and port. It will resolve link-up.com through DNS and expect a valid TLS connection for that domain. A local process on the user’s machine cannot casually take over that URL without DNS/hosts-file tricks, a trusted certificate setup, or a native container controlling the whole experience. Regarding web view UI performance, does having a full browser engine rendering the UI slow things down?

No, not automatically.

A WebView has overhead, but for Link-Up’s UI shape it should be fine. Profiles, grids, chat, settings, forms, filters, onboarding, modals, and map controls are exactly the kind of UI browser engines are good at. Apple explicitly describes WKWebView as a native view for presenting HTML, CSS, and JavaScript inside an app’s UI. (developer.apple.com)

The main costs are:

text
id="cfayzg"
cold startup
memory use
large DOMs
too much frontend JS
layout thrashing
huge images
bad scrolling/list rendering
over-rendering reactive components

The rendering itself is usually GPU-accelerated on modern platforms. Android’s rendering pipeline has supported hardware acceleration for normal View drawing for a long time, though it uses more RAM. (developer.android.com)

For Link-Up, the biggest performance win is actually your architecture:

text
id="0i7dpj"
static assets are local
API is local
database is local
media cache is local
frontend is thin
backend owns state

That removes the worst web-app performance problems: network latency, giant JS bundles, remote API chatter, and frontend state bloat.

So the rule should be:

text
id="85ddvn"
Use WebView for UI.
Do not make the WebView the app runtime.
Do not make the frontend heavy.

Native UI would outperform WebView for very high-frequency animations, huge infinite lists, camera-heavy workflows, games, AR, or deeply platform-specific gesture work. But Link-Up is mostly information UI plus realtime updates. A disciplined WebView UI should be fast enough.

The map is probably the heaviest part. That is where you need to be careful: marker count, clustering, tile loading, viewport updates, and DOM overlays. But that problem exists whether the app is native or web.

My practical answer: WebView is not the bottleneck unless you let the frontend become bloated.
